현재 작업 계획의 빈틈을 체크합니다: $ARGUMENTS

## 체크 흐름

### 1. Implicit Requirements 체크
대상 작업에 대해 "보통 필요하지만 자주 빠뜨리는 것" 확인:

**API 엔드포인트**:
- [ ] 에러 응답 형식이 정의되어 있는가?
- [ ] 입력값 검증 규칙이 있는가?
- [ ] 목록 API에 페이지네이션이 고려되어 있는가?
- [ ] 호출 제한(rate limiting)이 필요한가?
- [ ] 요청/응답 로깅이 명시되어 있는가?

**인증/인가**:
- [ ] 토큰 만료 정책이 있는가?
- [ ] 토큰 갱신(refresh) 흐름이 정의되어 있는가?
- [ ] 로그아웃 메커니즘이 있는가?
- [ ] 로그인 시도 제한(brute force 방지)이 있는가?

**DB 모델**:
- [ ] 자주 쿼리되는 필드에 인덱스가 있는가?
- [ ] soft delete가 필요한가?
- [ ] created_at, updated_at이 있는가?
- [ ] 외래키, unique 등 제약조건이 있는가?

**프론트엔드**:
- [ ] 로딩 중 UI가 정의되어 있는가?
- [ ] 에러 시 UI가 정의되어 있는가?
- [ ] 데이터 없을 때(empty state) UI가 정의되어 있는가?

**공통**:
- [ ] 실패 시 롤백 계획이 있는가?
- [ ] 기존 기능과 호환성 이슈가 없는가?

### 2. "What If" 분석
계획을 세운 직후, 관점을 바꿔서 자기 계획을 공격:
- "이 계획에서 실패할 수 있는 부분은?"
- "사용자가 명시하지 않았지만 당연히 필요한 것은?"
- "이 기능과 기존 기능 사이에 충돌 가능성은?"

### 3. 결과 분류
- **must_have**: 이거 없으면 장애/보안 이슈 → 자동으로 계획에 포함
- **should_have**: 없어도 동작하지만 실무에서 거의 항상 필요 → 사용자에게 제안
- **nice_to_have**: 있으면 좋지만 지금 안 해도 됨 → 기록만

범위 제한: 최대 5개 제안, 원래 범위 대비 20% 이내
